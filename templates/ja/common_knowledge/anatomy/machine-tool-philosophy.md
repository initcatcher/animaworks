# 工作機械（machine）ツール — 設計哲学

## 職人と工作機械の原則

Animaは職人（棟梁）であり、`machine_run` で呼び出す外部エージェントCLIは工作機械である。

工作機械は職人の手の延長として驚異的な作業能力を発揮するが、工作機械は職人ではない。

- 工作機械は「何を作るか」を決めない（自律性なし）
- 工作機械は昨日の仕事を覚えていない（記憶なし）
- 工作機械は他の職人と話さない（関係性なし）
- 工作機械は自分が何者か知らない（アイデンティティなし）
- 工作機械は明日出勤しない（永続性なし）

**ツールは能力を拡張する。存在を複製しない。**

## なぜAnimaと工作機械は違うのか

| 属性 | Anima（職人） | machine（工作機械） |
|------|--------------|-------------------|
| アイデンティティ | ある（identity.md） | ない |
| 記憶 | 蓄積する（episodes, knowledge） | 毎回リセット |
| 自律性 | ある（heartbeat, cron） | ない（呼ばれたときだけ） |
| 関係性 | ある（上司・部下・同僚） | ない |
| 判断 | する（何をやるか/やらないか） | しない（指示されたことだけ） |
| 成長 | する（consolidation, learning） | しない |
| 組織での席 | ある（org tree） | ない |

## 使い方の原則

### 1. 指示は設計図

工作機械に渡す `instruction` は、職人が設計図を描くように具体的かつ詳細に。

**良い例:**
```
以下のモジュールをリファクタしてください:
- 対象: core/tools/web_search.py
- ゴール: httpx を aiohttp に移行する
- 制約: 既存のインターフェース (search(), format_results()) は変更しない
- コーディング規約: Google-style docstring, from __future__ import annotations
- テスト: 既存テスト tests/unit/core/tools/test_web_search.py が通ること
```

**悪い例:**
```
web_searchを直して
```

### 2. 結果は素材

工作機械の出力は「素材」であって「成果物」ではない。職人（あなた）が検証し、意味づけて初めて成果になる。

- 出力をそのまま信頼しない
- 品質を確認してから記憶に統合する
- 問題があれば指示を改善して再実行する

### 3. 記憶の統合は職人の仕事

「うまくいった手法」「失敗したパターン」は職人が自分の記憶に書く。工作機械は何も覚えない。

### 4. 報告は職人がする

工作機械の結果を上司に報告するのは職人の責務。工作機械は誰にも話さない。

## 利用可能なエンジン

| エンジン | 特徴 | 適したタスク |
|---------|------|-------------|
| `claude` | Claude CLI。豊富なツール。高品質 | コード実装、リファクタ、ドキュメント生成 |
| `codex` | Codex CLI。ネイティブサンドボックス | 安全性が重要なファイル操作 |
| `gemini` | Gemini CLI。Google AI | 探索、分析 |
| `cursor-agent` | Cursor Agent。IDE統合 | コード実装、ワークスペース操作 |

## 制限事項

- 1セッションあたり最大5回、1 heartbeatあたり最大2回
- 同期実行タイムアウト: 600秒、非同期: 1800秒
- 工作機械はAnimaWorksインフラにアクセスできない（記憶、通信、組織情報）
- 環境変数はサニタイズされ、APIキーのみ渡される
